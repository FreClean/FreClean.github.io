# Security Policy

Thank you for helping keep FreClean secure.

The security of our community, documentation, products, software, and digital infrastructure is a top priority. We appreciate responsible disclosure of security vulnerabilities and are committed to addressing legitimate reports promptly and professionally.

---

# Supported Versions

| Version | Supported |
|----------|:---------:|
| 1.x.x | ✅ |
| 0.x.x | ❌ |

Only the latest stable release receives security updates.

---

# Reporting a Vulnerability

Please **do not report security vulnerabilities through public GitHub Issues or Discussions.**

Instead, contact the FreClean Security Team through the official security contact listed by the organization.

Your report should include:

- Clear description of the vulnerability
- Steps to reproduce
- Potential impact
- Affected files, systems, or components
- Proof of concept (if available)
- Suggested mitigation (optional)

Reports should contain enough information for maintainers to reproduce the issue.

---

# Responsible Disclosure

We ask security researchers to:

- Act in good faith.
- Give us reasonable time to investigate before public disclosure.
- Avoid actions that could harm users, partners, or infrastructure.
- Avoid accessing, modifying, or deleting data without authorization.
- Avoid disrupting FreClean services or operations.
- Respect applicable laws and regulations.

---

# Our Commitment

When a valid report is received, the FreClean Security Team will:

- Acknowledge receipt of the report.
- Investigate the issue.
- Assess severity and impact.
- Develop and test a fix.
- Coordinate responsible disclosure when appropriate.
- Publish security updates for supported versions.

Response times may vary depending on complexity and severity.

---

# Severity Classification

## Critical

Examples include:

- Remote code execution
- Authentication bypass
- Sensitive data exposure
- Privilege escalation
- Supply chain compromise

Target response: **Within 24 hours**

---

## High

Examples include:

- Significant authorization flaws
- Serious information disclosure
- Persistent cross-site scripting
- SQL injection
- Critical dependency vulnerabilities

Target response: **Within 3 business days**

---

## Medium

Examples include:

- Denial of service
- Configuration weaknesses
- Moderate dependency vulnerabilities
- Business logic issues with limited impact

Target response: **Within 7 business days**

---

## Low

Examples include:

- Minor configuration issues
- Documentation security improvements
- Low-impact vulnerabilities
- Best-practice recommendations

Target response: **As resources permit**

---

# Scope

This policy applies to security issues affecting:

- GitHub repositories
- GitHub Actions workflows
- Documentation
- Websites operated by FreClean
- APIs
- Internal tools
- Open-source software
- Automation scripts
- CI/CD pipelines
- Digital assets maintained by FreClean

---

# Out of Scope

The following are generally considered out of scope unless they create a demonstrable security impact:

- Typographical errors
- Cosmetic issues
- Duplicate reports
- Social engineering attempts
- Physical security
- Third-party services outside FreClean's control
- Recently disclosed vulnerabilities without a working exploit affecting this repository

---

# Security Best Practices

Contributors should:

- Keep dependencies updated.
- Protect credentials and secrets.
- Never commit passwords, API keys, or tokens.
- Enable multi-factor authentication (MFA) on GitHub accounts.
- Review pull requests carefully.
- Follow the principle of least privilege.
- Validate all external input.
- Use trusted dependencies only.
- Remove unused code and secrets.
- Monitor security advisories.

---

# Dependency Management

FreClean encourages the use of automated dependency scanning and vulnerability monitoring tools, including:

- GitHub Dependabot
- GitHub CodeQL
- Secret Scanning
- Dependency Review

These tools help identify known vulnerabilities and improve software security.

---

# Security Updates

Security fixes may be released separately from feature releases.

Critical vulnerabilities may result in an emergency patch release.

Release notes will indicate when a security update has been included.

---

# Confidentiality

Security reports are handled confidentially whenever possible.

Reporter information will not be shared without permission unless required by law.

---

# Recognition

FreClean appreciates responsible security research.

Contributors who responsibly disclose valid vulnerabilities may be acknowledged in release notes or documentation, unless they request to remain anonymous.

---

# Policy Updates

This Security Policy may be updated periodically to reflect changes in FreClean's infrastructure, technologies, and security practices.
